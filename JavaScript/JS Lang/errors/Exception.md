


### Zero-cost exceptions


Short answer: **“Zero-cost exceptions” have _low cost when not used_, but a _high cost when an exception is actually thrown_.**

Here’s what that really means 👇

---

## What “zero-cost exceptions” means

In languages like **C++** (and partly Rust with panics/unwinding):

- ✅ **No runtime overhead in the normal path**
    
    - No extra checks
        
    - No bookkeeping per function call
        
- The compiler assumes _nothing goes wrong_
    

So when **no exception is thrown**, the cost is essentially **zero**.

---

## Why the cost is high _when thrown_

When an exception _does_ happen:

1. **Stack unwinding**
    
    - Walks back through many stack frames
        
    - Runs destructors / cleanup code
        
2. **Metadata lookup**
    
    - Uses exception tables generated by the compiler
        
3. **Cache-unfriendly control flow**
    
    - Jumps across many frames
        
    - Breaks branch prediction
        

👉 This makes throwing an exception **orders of magnitude slower** than:

- returning an error value
    
- using a `Result` / `Either` type
    

---

## Cost comparison (rough intuition)

|Operation|Relative cost|
|---|---|
|Normal function return|1×|
|Error via return value|~1–3×|
|Zero-cost exception thrown|**100–1000×**|

(Exact numbers vary by platform, but the _shape_ is correct.)

---

## Why languages still use them

Zero-cost exceptions are great when:

- Errors are **rare**
    
- You want **clean, linear code**
    
- Performance of the _success path_ matters most
    

That’s why they’re common in:

- C++
    
- some system-level code
    
- performance-critical libraries
    

---

## Contrast with other models

### Java / JavaScript

- Exceptions have **constant overhead**
    
- Try/catch adds cost even when no exception is thrown
    
- Throwing is still expensive, but less extreme than C++
    

### Rust / Gleam / Elixir

- Prefer **explicit error values**
    
- Cost is predictable
    
- No hidden control flow
    

Example (Gleam / Rust style):

```text
Result(value, error)
```

---

## Key takeaway

> **Zero-cost exceptions optimize for the common case, not the failure case.**

- ✔ Fast when nothing goes wrong
    
- ❌ Very expensive when something does
    
- ❌ Bad for control flow or frequent errors
    

---

### Rule of thumb

- **Rare, exceptional errors** → zero-cost exceptions are fine
    
- **Expected or frequent errors** → use return values / Result types
